## 今天做完的

### 路由
- **iBGP**（core1/core2，AS 65001）：基于 Loopback 对等，OSPF 提供可达性，叠加不抢转发（iBGP AD 200 > OSPF 110）。
- **eBGP**（edges ↔ inet，AS 65001 ↔ 65100）：把 inet 用 **FRR** 变成 ISP 路由器，宣告 `0.0.0.0/0` + 假互联网前缀（203.0.113/198.51.100，RFC5737）；edges 宣告园区网段。踩坑：FRR 8.x 的 **RFC 8212** 默认拒绝（`no bgp ebgp-requires-policy`）；EOS **eBGP AD=200**（非 Cisco 的 20）。

### 可观测性
- **BGP 监控上线**：gNMI 订阅 session-state（on-change）+ prefix 计数（sample）。session-state 是 6 态 FSM，映射成 `1..6`（IDLE..ESTABLISHED），**不压成 1/0**——现场演示证明价值：关邻居时 core1=Idle(1) / core2=Active(3) 两端不同态，能分清"谁关的 vs 谁连不上"。Grafana 加了 2 个面板（状态时间线 + 前缀数）。
- **修基数泄漏**：每进程遥测 × rsyslog pid churn × 6h 过期 → 序列累积到 ~4000 还在涨。修法：`expiration 6h→60s`（截断累积，活跃序列封顶 ~1435 稳定）+ 看板 `proc=~"[A-Z].*"` 只显真 EOS agent。proc_info 3949→364。

### 排障（三个案例，方法论闭环）
- **幻影内存尖峰（13:52）**：日志全空 → 时间序列定位 ConfigAgent+Sysdb 同步 +54MB → 实验证伪 CLI 读、证实 **gNMI 批量读**（读操作不留日志，只有写才记）。→ 已写成 `diag_case/diagnosis.md`。
- **rsyslog 崩溃循环 248 万次**：systemd socket-activation 与 rsyslog 抢 `/run/systemd/journal/syslog`，EADDRINUSE + 限流关闭 → 每 100ms 重启不止。无害但制造 pid churn（正是上面基数泄漏的源头）。
- **18:35 内存/CPU 抬升**：定位为 BGP agent（Bgp-main，四台合计 ~1.5G）启动的正常开销 + 配置验证活动，非泄漏。

### 治理设计（inventory + 操作管理）— 另一会话产出
> 背景：发现有 2 个其他 agent 也在读/改这些设备，查询与配置无人管理，现象分析难闭环，且无资源清单。
- 写了 `mgmt/design.md`（中英双语，未提交/未推）——补全两项管理的设计，暂不含实现代码。
- **核心洞察**：读操作不进设备日志 → 审计**必须在 agent 发起端做**；清单与审计是同一个"接入层"的两面。前提已确认：三个 agent 都可控 → choke point 用**共享 MCP server 自愿收敛**即可，无需强制代理。
- **三层防护**：① 接入层 MCP（`dev.read/write/exec` + `agent_id`/`correlation_id` + 写前**租约**防互踩）；② config-hash 哨兵（兜底，捕获一切写，哪怕绕过接入层）；③ 闭环 = audit→Loki→**Grafana annotations**，把 §45 那条"指标+日志+变更三线合一"落地。
- **新增亮点 §4.4 读的限流/配额 = agent 层的 CoPP**：论证"读不是无害旁观"——打的是控制平面，失控会饿死 BGP/OSPF/BFD（`diag_case` 尖峰即最小样本）。机制：批量读串行/限速、每设备配额、重路径(大表/show tech)禁 agent 拉走遥测、全局读预算、限流事件也记审计。
- 分阶段：P0 清单+哨兵（当天可用）→ P1 `campusctl` 封装 → P2 MCP+Grafana → P3(可选) 凭据封闭。
- 与 Ansible 关系（讨论未落文档）：Ansible 能吃掉 inventory + 写的批量下发（`arista.eos`），**吃不掉跨 agent 的读审计/协调/闭环**——后者仍需接入层。TODO：可补 design.md §9。
